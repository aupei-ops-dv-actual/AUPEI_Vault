## Phase 1: Reckoning — What Actually Happened

### Operator Framework Findings
- Attempted deployment of Clerk of Constraint as OpenJarvis custom operator (TOML manifest, [operator.agent] schema).
- **qwen2.5:14b capability ceiling confirmed:** Model can retrieve (memory_search via jarvis ask works) and triage simple prompts. Cannot execute multi-step tool-calling protocols in operator context.
- Built-in operators (knowledge_curator) work after daemon restart. Custom operators with complex prompts fail silently.
- knowledge_curator finds "no unprocessed memories" — designed for agent artifacts, not indexed governance docs. Design mismatch, not bug.

### Infrastructure Cascade
- NFS symlinks for agents.db, telemetry.db, traces.db caused stale state — removed, replaced with local placeholder DBs.
- Tilde path expansion bug in config.toml: Python sqlite3.connect() doesn't expand `~`. Fixed with sed to absolute paths.
- Daemon restart (`sudo launchctl unload/load`) was critical final fix — jarvis serve was caching old config from boot.
- `jarvis operators logs` broken (SchedulerStore.get_runs missing). Non-fatal.

### Architectural Pivot: Bailiff Pattern
- R@'s insight: "CoC is a pre-ingestion operator. We don't need to and shouldn't ask it to reason through law — just a first pass/sort/correct. Like a doorman or court bailiff."
- **Solution:** 13-line shell script (`coc_bailiff.sh`) bypasses operator framework entirely. Reads file via `cat`, passes content to `jarvis ask` with simple question list.
- Mortal by design: born per audit, dies after report. Amnesia by mortality, not by tool restriction.
- First audit: 7/10 clean, 3 arguable, 0 wrong. **Operational.**
